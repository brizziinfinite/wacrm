- **Checagem de assinaturas reais antes de codar**: pra Task 4, delegado a um subagente Explore pra

- **Verificação empírica de crypto antes de declarar pronto**: correção do gap 1 exigiu reimplementar

  mockam fetch/decrypt). Escrito teste de round-trip real contra o `encrypt()` de produção — 1ª tentativa
  falhou (bug real no teste: env var lida no module top-level antes do `beforeAll` setar), corrigido e
  confirmado passando. Sem esse teste, o bug ficaria invisível até prod (falha silenciosa via try/catch).

### Problemas resolvidos

- **`sendText()` engolia falha de envio e reportava "sent" mesmo quando Meta/Instagram API
  retornava erro** (`process-ai-messages`, `qualify-lead`) — regressão vs. código antigo que checava
  `response.ok` e retornava 500. Corrigido: `status` agora reflete `metaMessageId` real
  (`sent`/`failed`); `process-ai-messages` retorna 502 se envio falhar.
- **Channel-routing só checava `external_id` truthy, não exclusividade de canal** — contato
  Instagram com `external_id` nulo caía no envio WhatsApp com o placeholder `instagram:<igsid>`
  como telefone, gerando request lixo pra API errada. Corrigido nos 4 pontos (`process-ai-messages`,
  `qualify-lead`, e os dois `sendText`/`sendViaMeta` em `src/lib`).
- **`engineSendMedia`/`sendInteractiveViaMeta` (flows) nunca tiveram branch Instagram** — ao
  contrário de `engineSendText`, que já tratava o canal. Contato Instagram com node `send_media` ou
  interativo caía direto no fluxo WhatsApp com o IGSID como telefone. Corrigido: erro claro
  ("Instagram does not support media/interactive sends from flows") em vez de fallback silencioso.
- **Settings do Instagram buscava `access_token`/`verify_token` cifrados pro browser**, nunca
  renderizados — exposição desnecessária de ciphertext no devtools/network tab. Removido do
  `select()`, tipo local via `Omit`.
- **Token de acesso Instagram na URL como query param** em vez de header `Authorization` (padrão
  usado no lado WhatsApp) — risco de vazamento em logs de proxy/CDN. Corrigido nas duas funções de
  `graph-api.ts`.
- **Duplicação**: bloco de envio Instagram (~35 linhas) copiado entre `automations/meta-send.ts` e
  `flows/meta-send.ts` — extraído `sendInstagramTextAndLog` em `graph-api.ts`, ambos os arquivos
  agora chamam o helper. Mesma duplicação no lado edge functions (`qualify-lead` e
  `process-ai-messages` tinham `sendWhatsAppText`/`sendInstagramText`/`decryptToken`/`hexToBytes`
  idênticos) — extraído `supabase/functions/_shared/meta-send.ts`; comentário antigo alegando que
  "Deno edge functions não compartilham módulo fora do próprio diretório" estava errado (repo já
  usa `_shared/llm-retry.ts` em outras functions).
- **`findExistingContactByExternalId` fazia `select("*")`** no hot path de todo DM Instagram
  inbound — trocado por `id, channel, external_id` (únicos campos usados pelo único caller).

### Decisões técnicas

- Erro explícito > fallback silencioso: Instagram não suporta mídia/interativo via essa API — em

- Extração dos helpers compartilhados ficou no módulo já canônico de cada domínio
  (`instagram/graph-api.ts` pro lado Node, novo `_shared/meta-send.ts` pro lado Deno) em vez de
